# Contributing Guidelines

Before you contribute to GD Resources, there are a few things you should know:

## How to setup GD Resources?

### Prerequisites

- <Link href="https://nodejs.org/en/download">Node.js ≥20.9.0</Link>
- <Link href="https://git-scm.com/install">Git</Link>
- A computer running Windows, macOS, or Linux (you can use Termux if on Android - but it's annoying)

### Initial setup

<Steps>
  <Step>
    Clone the repository:

    ```bash
    git clone https://github.com/OmgRod/gd-resources
    ```
  </Step>
  <Step>
    Navigate to the new folder:

    ```bash
    cd gd-resources
    ```
  </Step>
  <Step>
    Install all packages:

    ```bash
    npm install
    ```
  </Step>
  <Step>
    Start the server

    ```bash
    npm run dev
    ```
  </Step>
</Steps>

### How do components work here?

GD Resources is based on a docs template made by OmgRod called <Link href="https://miniwiki.omgrod.me/">MiniWiki</Link>. To find out how components work and how to use them, read <Link href="https://miniwiki.omgrod.me/guides/components/index">here</Link>.

## How to give yourself credit?

<Steps>
    <Step>
        We use GitHub to host this website's code, so you should <Link href="https://github.com/signup">create an account</Link> if you haven't yet.
    </Step>
    <Step>
        Add yourself as an author in `content/authors.json`, like this:
        ```json
        {
            "YourUsername": {
                "name": "YourUsername",
                "avatar": "YourAvatarUrl", // you can upload your own avatar to `public/assets/avatars`
                "url": "YourPortfolioOrProfile.com"
            }
        }
        ```
    </Step>
    <Step>
        If you are creating a new category/subcategory, ensure it contains **at least** the following files:
        
        1. `index.mdx`
        2. `guides/index.mdx`
    </Step>
</Steps>

## Category file templates

### `index.mdx`

```mdx
---
title: [CATEGORY NAME] - Table of Contents
description: [CATEGORY NAME] tutorials
editPage: true
---

# [CATEGORY NAME]: Table of Contents

<Link href="/[CATEGORY PATH]/guides/index">View official guides here</Link>

## Useful Resources

<Table>
  <THead>
    <TR>
      <TH>Resource</TH>
      <TH>Developer</TH>
      <TH>Description</TH>
    </TR>
  </THead>
  <TBody>
    <TR>
      <TH><Link href="[RESOURCE URL]">[RESOURCE NAME]</Link></TH>
      <TH><Link href="[DEVELOPER GITHUB/PORTFOLIO]">[DEVELOPER NAME]</Link></TH>
      <TH>[DESCRIPTION]</TH>
    </TR>
  </TBody>
</Table>
```

You may add *more* rows by cloning the `<TR>` tag and pasting it below the closing `</TR>` tag.

### `guides/index.mdx`

```mdx
---
title: [CATEGORY NAME] - Guides
description: [CATEGORY NAME] official guides
editPage: true
---

# [CATEGORY NAME]: Guides

## Sub-categories

<GuidesTable subdirectory="[CATEGORY PATH]/guides/[SUBCATEGORY PATH]" caption="[SUBCATEGORY NAME]" />

## Individual Guides

<GuidesTable guides={guides} />
```

Sub-categories are **optional**. If no sub-categories are present, remove the two subtitles and the `GuidesTable` in between.

## Creating Guides

You can create a guide using the following template:

```mdx
---
title: [GUIDE TITLE]
description: [GUIDE DESCRIPTION]
author: [YOUR USERNAME]
editPage: true
order: ORDER_NUMBER
---

# [GUIDE TITLE]

ADD YOUR STUFF HERE
```

It is important that:

1. The guide title is the exact same in both fields.
2. You keep the description relatively short (1-2 sentences max). But keep it descriptive enough as this will appear in `category/guides/index.mdx`.
3. Your username is the exact same as the one you put in `authors.json`.
4. `editPage` is ALWAYS TRUE!
5. `ORDER_NUMBER` must be a number. The higher the number, the further down it appears. It is completely optional and only recommended for subcategory guides.

Also, whenever you create a guide, you must add it to `sidebar.json`, like this:

```json
{
  "title": "CATEGORY TITLE",
  "collapsible": true,
  "items": [
    { "title": "Table of Contents", "path": "/CATEGORY PATH/index" },
    { "title": "Official Guides", "path": "/CATEGORY PATH/guides/index" },
    { "title": "SUBCATEGORY/SERIES NAME", "collapsible": true, "items": [
      { "title": "1: PART 1 TITLE", "path": "/CATEGORY PATH/guides/SUBCATEGORY PATH/chap1" }
    ]},
    { "title": "YOUR GUIDE NAME (IF IN SAME CATEGORY)", "path": "/CATEGORY PATH/guides/index" },
  ]
},
```

In the JSON code above, there is an example for regular guides, and an example for subcategories (or a series of tutorials). Below are the rules for either of the guides:

<TabPanels>
  <Tab title="Regular Guides">
    1. Always place new guides **after** the two default entries: - Table of Contents - Official Guides
    2. Each guide must have a unique, readable title (avoid generic names like “Guide 1”).
    3. Keep paths consistent with folder structure:
      - `/CATEGORY PATH/guides/guide-name`
    4. Do not create unnecessary nesting if the guide is standalone.
    5. Use lowercase, hyphen-separated file paths for consistency.
  </Tab>
  <Tab title="Subcategories">
    1. Subcategories are used for multi-part tutorials or structured learning paths.
    2. Each part must follow a strict naming format:
      - `[Part X]: Short descriptive title`
    3. Ensure chapter order is explicit (1, 2, 3...) and never rely on alphabetical sorting.
    4. All parts must live inside the same subcategory folder.
    5. Avoid mixing standalone guides and series content inside the same subcategory.
    6. The first part should clearly introduce the series and explain its purpose briefly.
  </Tab>
  <Tab title="General Rules">
    1. Keep titles short, descriptive, and consistent across sidebar and frontmatter.
    2. Ensure every path matches an actual file — broken links are not allowed.
    3. Never duplicate entries in the sidebar.
    4. If unsure where a guide belongs, default to a regular guide instead of creating a new subcategory.
    5. Keep the sidebar clean - prefer flat structure unless hierarchy is truly needed.
  </Tab>
</TabPanels>

## Pushing your changes to GitHub

Now that you finished writing your guide, you may push your changes to GitHub:

<Steps>
    <Step>
        Commit your changes and push them once you're done:

        ```bash
        git commit -m "what did you change?"
        git push
        ```

        <Link href="https://code.visualstudio.com/Download">VS Code</Link> is highly recommended for Git operations if you're new to Git.
    </Step>
    <Step>
        Once your changes have been pushed on GitHub, open the repository (should be in your account). Click the button that says "Contribute" and when opened, click "Open pull request". You should see something like shown below (this is on a different repository so this will look slightly different for you):

        <Image
          src="/assets/contribute.png"
          alt="A popup that shows up when clicking the 'contribute' button"
        />
        <Image
          src="/assets/pr.png"
          alt="The pull request creation screen"
        />
    </Step>
</Steps>
